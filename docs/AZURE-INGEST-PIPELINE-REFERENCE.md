# Azure Ingest Pipeline Reference

This document describes the ingest pipeline architecture for Azure services in Cloud Loadgen for Elastic.

## Overview

Each Azure generator produces documents in **vendor-native format** (matching Azure Monitor / Activity Log / Diagnostic Log schemas). Ingest pipelines handle all **ECS normalization** — mapping Azure-native fields to Elastic Common Schema fields.

## Pipeline registry

All Azure pipelines are generated from the service catalog and stored in:

```
installer/azure-custom-pipelines/pipelines/registry.mjs
```

The registry is generated by `npm run gen:azure-pipelines` (via `scripts/generate-azure-pipeline-registry.mjs`) and exports a `PIPELINE_REGISTRY` array.

## Pipeline naming convention

| Pattern                           | Example                                 | Purpose                           |
| --------------------------------- | --------------------------------------- | --------------------------------- |
| `logs-azure.{dataset}-default`    | `logs-azure.activitylog-default`        | Log parsing and ECS normalization |
| `metrics-azure.{dataset}-default` | `metrics-azure.virtualmachines-default` | Metric field mapping              |

## Standard processors

Every Azure log pipeline includes these processor stages:

### 1. ECS normalization (`ecsNorm`)

Maps Azure-native fields to ECS equivalents:

| Azure native field      | ECS field                                                   |
| ----------------------- | ----------------------------------------------------------- |
| `operationName`         | `event.action`                                              |
| `category`              | `event.category` (mapped)                                   |
| `level`                 | `log.level`                                                 |
| `resultType`            | `event.outcome` (Success/Succeeded = success, else failure) |
| `resourceId`            | `cloud.instance.id`                                         |
| `properties.statusCode` | `http.response.status_code`                                 |

### 2. Identity extraction (`azureIdentityExtract`)

| Azure native field                                                          | ECS field                 |
| --------------------------------------------------------------------------- | ------------------------- |
| `identity.claims.http://schemas.xmlsoap.org/ws/2005/05/identity/claims/upn` | `user.email`, `user.name` |
| `callerIpAddress`                                                           | `source.ip`               |
| `identity.claims.appid`                                                     | `user_agent.name`         |

### 3. Related fields (`populateRelatedFields`)

Populates `related.ip` and `related.user` arrays from extracted identity fields.

### 4. Vendor cleanup (`vendorCleanup`)

Optionally removes Azure-native fields that have been mapped to ECS equivalents.

## Installing pipelines

```bash
# Via wizard
# Setup → Install Cloud Loadgen Integrations (Azure)

# Via CLI
npm run setup:azure-pipelines

# Via loadgen packs (per-service)
npm run setup:azure-loadgen-packs
```

## Regenerating the registry

If you add new Azure generators or modify field mappings:

```bash
npm run gen:azure-pipelines
```

This reads the Azure elastic maps and generator metadata to produce the pipeline registry.
